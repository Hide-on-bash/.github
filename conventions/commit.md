# 커밋 컨벤션 가이드

> 이 문서는 팀 프로젝트에서 Git 커밋 메시지를 일관되게 작성하기 위한 규칙입니다.
> 모든 팀원이 이 규칙을 따라야 합니다.

---

## 왜 커밋 컨벤션이 필요한가?

- 나중에 git log를 봤을 때 **무슨 작업을 했는지 바로 파악** 가능
- 버그가 생겼을 때 **어느 커밋에서 문제가 생겼는지** 빠르게 추적 가능
- 팀원 간 **코드 리뷰가 수월**해짐

---

## 기본 구조

```
<타입>: <제목>

<본문 - 선택사항>
```

### 예시

```
feat: 사용자 로그인 기능 구현

- 이메일/비밀번호 입력 유효성 검사 추가
- 로그인 성공 시 JWT 토큰 발급
- 실패 횟수 초과 시 계정 잠금 처리
```

---

## 타입 (Type)

모든 커밋 메시지는 아래 키워드 중 하나로 시작하며, 뒤에 콜론(`:`)과 한 칸 공백을 붙입니다.

| 타입 | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 추가 | `feat: 데이터 분석 모듈 구현` |
| `fix` | 버그 및 오류 수정 | `fix: 입력값 빈 문자열 검증 누락 수정` |
| `docs` | 문서 수정 (README, 명세서 등) | `docs: README 설치 가이드 섹션 추가` |
| `refactor` | 코드 리팩토링 (기능 변화 없음) | `refactor: DB 연결 객체를 utils.py 싱글턴으로 중앙화` |
| `test` | 테스트 코드 추가 또는 수정 | `test: 입력 유효성 검증 단위 테스트 추가` |
| `chore` | 빌드 설정, 패키지 의존성 수정 | `chore: pandas 의존성 버전 고정` |

---

## 제목 (Subject) 작성 규칙

| 규칙 | 설명 |
|------|------|
| **50자 이내** | 제목이 너무 길면 git log에서 잘림 |
| **마침표 없음** | 제목 끝에 `.` 붙이지 않음 |
| **평서문 형식** | "무엇을 했는지"를 명확하게 |
| **한국어 사용** | 팀 내 공통 언어로 작성 |

---

## 본문 (Body) 작성 규칙

제목만으로 설명이 부족할 때 작성합니다. **선택사항입니다.**

- 제목과 **빈 줄 하나**로 구분
- **불릿 포인트(`-`)** 로 작성
- **무엇을, 왜** 변경했는지 설명
- 한 줄은 72자 이내

```
fix: API 응답 파싱 예외 처리 누락 수정

- 빈 응답 반환 시 KeyError 발생 케이스 확인
- 응답 접근 전 None 체크 로직 추가
- 동일 버그 재발 방지를 위해 utils.py에 중앙화
```

## 좋은 커밋 vs 나쁜 커밋

### ✅ 좋은 커밋 메시지

```
feat: 검색 필터 3단계 분기 로직 구현
fix: 입력 검증 누락으로 None 전달되는 문제 수정
refactor: UserState TypedDict 필드명 전면 rename
docs: README 모듈 역할 명세 추가
```

### ❌ 나쁜 커밋 메시지

```
수정                   ← 무엇을 수정했는지 알 수 없음
fix                    ← 어떤 버그인지 불명확
작업중                 ← 작업 상태가 아닌 변경 내용을 써야 함
여러 가지 수정         ← 여러 작업을 하나의 커밋으로 묶으면 안 됨
WIP                    ← 공유 브랜치에는 미완성 커밋 금지
```

---

## 커밋 단위 원칙

> 하나의 커밋 = 하나의 논리적 변경 사항

| 원칙 | 이유 |
|------|------|
| 하나의 변경 이유만 담기 | 나중에 특정 변경만 되돌릴 때 편함 |
| 기능 완성 후 커밋 | 실행 안 되는 상태로 커밋하면 다른 팀원에게 피해 |
| 작업 중간 저장은 로컬에서만 | WIP 커밋을 공유 브랜치에 push하지 않기 |

## 절대 하면 안 되는 것 🚫

| 금지 | 이유 |
|------|------|
| `.env`, `secrets.json`, 비밀번호·API 키 커밋 | GitHub에 올라가면 삭제해도 히스토리에 영구 보관. 보안 사고로 이어짐 |
| 수백 MB 이상의 이미지·영상·모델 파일 커밋 | 리포지토리 크기 폭증. 클론 시간 길어짐 |
| `__pycache__/`, `.DS_Store`, `node_modules/` 커밋 | `.gitignore`로 처음부터 제외 |
| 의미 없는 메시지(`수정`, `fix`, `1`) 작성 | 팀원이 히스토리 파악 불가 |
| 여러 기능을 한 커밋에 묶기 | 코드 리뷰 어렵고, 특정 변경만 되돌리기 불가 |
| 공유 브랜치에 직접 커밋 | PR 없이 코드가 들어가면 리뷰 불가 |

## 커밋 전 체크리스트

```
□ 하나의 커밋에 하나의 논리적 변경 사항만 담겼는가?
□ 제목만 보고 변경 내용을 유추할 수 있는가?
□ 파일 변경 내용과 메시지 의도가 일치하는가?
□ .env, API 키 등 민감 정보가 포함되지 않았는가?
□ 불필요한 공백이나 테스트용 print/로그가 남아있지 않은가?
□ 코드가 실행 가능한 상태인가?
```
